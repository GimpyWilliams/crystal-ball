1 · What it is

The crystal-ball MCP server reads a live fort over the DFHack RPC socket (127.0.0.1:5000) and answers operational questions about any production pipeline — not just beer, which was the original focus. It is overwhelmingly read-only: it diagnoses why an industry is stalled, audits stock and storage, profiles dwarves, and (new) looks up artifacts, deities and worship.

Transportstdio — the MCP client launches it as a subprocess; it binds no network port.
Only egressthe loopback DFHack RPC socket. The client refuses any non-loopback host.
RequiresDF + DFHack running with a fort loaded; Python 3.13 venv at agent/.venv.

2 · How a tool works (the repeating pattern)

Every feature is the same four-layer slice. Adding a capability means following it end to end — no framework magic.

MCP tool (mcp_server.py)        @mcp.tool() foo_report / foo_data
        │
Python fetch + format (foo.py)  fetch_foo() -> dict   format_foo(dict) -> str
        │
run_intel() (intel.py)          prepend _prelude.lua, send over RPC, parse JSON
        │
Lua query (scripts/foo.lua)     read df.global.world.*, section() per block, finish()
  • Read path: run_intel("foo.lua", [args]) runs RunCommand("lua", [prelude+script, …args]); the script print()s one JSON blob that Python parses.
  • Twin tools: each capability ships a _report (human text) and a _data (raw JSON) variant from the same fetch.
  • Graceful degradation: section(name, fn) wraps each block in pcall; a renamed DF field downgrades that section to an errors[] note instead of breaking the whole report.
  • Fort gate: every script sets fort_loaded = (world.map.block_index ~= nil) and bails early at the main menu.

4 · Safety model — why it's trustworthy

  • Loopback only. The client rejects any host other than 127.0.0.1/localhost/::1.
  • No caller Lua. Only the bundled, audited scripts/*.lua files ever run; selector/labor args are forwarded as data, never executed.
  • No open port. stdio transport; nothing listens.
  • Writes are explicit & reversible. Gated behind apply+confirm, validated, and described as undoable in-game.
  • Fails soft. Per-section pcall + a top-level connection guard turn DFHack hiccups into messages, not crashes.

5 · Key technical findings

Lessons that cost real debugging time and should shape every future query.

⚠ The RPC is single-threaded — a slow query poisons the server

A client-side timeout does not cancel the work: DFHack keeps running the Lua loop, blocking new connections (even the handshake) until it drains. The default socket timeout is 5s.

Lesson: bound the work inside Lua. Filter cheaply before any expensive call, and never iterate a worldgen-scale collection doing per-item RPC-heavy work.

⚠ Artifact scale: 22,311 records, all with resident items

This world's world.artifacts.all holds tens of thousands of (historical) artifacts, and DF keeps every artifact's .item in memory — so neither a.item ~= nil nor world.items.all membership tells you which are in your fort. Translating all 22k names blew the timeout.

Fix: filter on the item's placement flags (on_ground / in_building / in_inventory) before translating — cut 22,311 → 25 instantly. Name-translation is the expensive operation.

⚠ Two name forms — dwarvish vs English

translateName(name) gives the romanized dwarvish the player sees ("Zodost Mogshum"); translateName(name, true) gives the English meaning ("The Rot of Elbows"). Searching the wrong form silently finds nothing.

Lesson: match against both forms; display dwarvish primary, English in parentheses.

⚠ DFHack struct paths rarely match the obvious guess

7 · Takeaways at a glance

What works well

  • One consistent four-layer pattern — trivial to extend.
  • Strong, layered safety: loopback, no caller Lua, gated writes.
  • Diagnosers give ranked root causes, not just raw data.
  • _report/_data twins serve both chat and tooling.
  • Per-section pcall survives DF version drift.

What to watch

  • Single-threaded RPC + 5s timeout = slow queries are dangerous.
  • Worldgen-scale collections need cheap pre-filters.
  • DFHack field paths drift and surprise — probe, don't assume.
  • CLI lags MCP in coverage.
  • Naming still reflects the beer-only origin.
